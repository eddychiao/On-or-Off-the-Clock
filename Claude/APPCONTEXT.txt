  ---
  Agent Context Prompt

  Stack: React 19 + TypeScript 6, Vite 8, React Router 7 (HashRouter), Supabase JS v2. No UI library — all styles are hand-written CSS. npm run dev to start, npm run build to verify (runs tsc -b then Vite).

  ---
  Architecture

  src/
    types/index.ts        — all shared TS interfaces and type constants
    utils/                — pure helper functions, no side effects
    lib/supabase.ts       — single exported supabase client (reads VITE_SUPABASE_URL/ANON_KEY)
    lib/storage.ts        — one async function per CRUD op, organized by table
    context/AuthContext.tsx  — AuthProvider + useAuth()
    context/AppContext.tsx   — AppProvider + useApp(); holds all app state
    components/           — shared UI components, each paired with a same-name .css file
    pages/                — one file per route, each paired with a same-name .css file
    App.tsx               — provider tree + HashRouter routes
    index.css             — global design tokens, layout primitives, shared utility classes

  ---
  Supabase pattern

  One async function per operation, throws on error:

  export async function getItems(): Promise<Item[]> {
    const { data, error } = await supabase
      .from('items')
      .select('*')
      .order('created_at', { ascending: false });
    if (error) throw error;
    return (data ?? []).map(parseItem);
  }

  Numeric quirk: numeric(10,2) columns come back as strings. Every fetch passes results through a parseX() function that casts numeric fields back with Number(). Always do this for new numeric columns.

  Never .catch() inside storage functions — let the caller handle errors. Never import storage directly from a component or page — always go through context.

  ---
  State management pattern

  AppContext loads all data once on mount with Promise.all. Mutations apply optimistic local updates first, then persist in the background:

  const addItem = useCallback(async (item: Item) => {
    setItems(prev => [item, ...prev]);
    await storage.addItem(item);
  }, []);

  New entity types follow the same pattern: state, load in Promise.all, three useCallback mutations (add/update/delete).

  Auth uses Supabase email+password. useAuth() exposes { user, signIn, signOut }. Supabase RLS controls write access — never implement access control in frontend code, only gate UI elements based on user.

  ---
  CSS conventions

  Design tokens are CSS custom properties in index.css. Always use them — never hardcode colors, radii, shadows, or transition durations.

  Every component and page has its own .css file imported directly in its .tsx file. Scope styles with the component's root class. All measurements in px or %. Flexbox is the standard layout tool.

  Define reusable global classes (buttons, cards, form elements, layout primitives) in index.css. Write component-scoped CSS only for what can't be shared.

  ---
  Component conventions

  - One default export per file, same name as the file.
  - Props interface declared inline above the component: interface FooProps { ... }.
  - Internal sub-components used only within one file live below the main export as named functions — don't export them.
  - useState for local UI state, context for shared app state, useMemo for derived data from large arrays.
  - No comments unless the reason is non-obvious. No docstrings.
  - IDs generated with crypto.randomUUID().
  - All types and interfaces live in src/types/index.ts, not inline in component files.

  ---
  Adding a new feature — checklist

  1. New data type → interface in types/index.ts, CRUD in storage.ts (with parseX() for numeric fields), state + mutations in AppContext.
  2. New page → pages/MyPage.tsx + MyPage.css, route in App.tsx, nav entry in NavBar.tsx. Wrap content in the shared .page class.
  3. New component → components/MyComponent.tsx + MyComponent.css. Use global classes before writing component-scoped CSS.
  4. Verify → npx tsc --noEmit (zero errors), then npm run build.
